Defensive Pedal — Design Audit
SCOPE apps/mobile DATE 2026-04-18 VERSION v1.0 tokens
SWOT · Constructive critique · Prioritised recommendations

A strong safety-first system, undermined by uneven adoption across screens.

The token layer, theme provider and atomic component library are well-conceived and safety-literate. The friction sits one layer up: screens routinely bypass the system — hardcoded hex values, inline rgba(), re-implemented toggles, and direct token imports that defeat theming. The result is a design that looks coherent from five metres and increasingly inconsistent up close.

01

SWOT analysis

Evidence-based

Grounded in the code in apps/mobile/src/design-system/ and the 31 screens under apps/mobile/app/. Each quadrant is backed by specific files and counts — not vibes.

Positive · Internal

Strengths

What is genuinely working
  • S1
    Semantic safety palette is disciplined. safe / caution / danger / info are kept separate from brand colour and documented as "never decorative". This is exactly the right instinct for a life-safety product.
  • S2
    Coherent token layer. Typography scale, 4-px spacing, platform-aware shadows, motion with reduced-motion hooks, z-index and icon-size tokens — the taxonomy is almost complete.
  • S3
    Clean atomic architecture. 25 atoms, 18 molecules, 29 organisms with TypeScript props and JSDoc headers. Accessibility is pre-wired into Button, Badge, BottomNav.
  • S4
    Domain-specific organisms. NavigationHUD, RouteComparisonPanel, HazardAlertPill, ElevationProgressCard — these are not a generic kit dressed in yellow; they actually encode the product's safety logic.
  • S5
    Glanceability rules are codified. Forced dark mode during navigation, monospace for distances and ETAs, haptic suppression while riding. These are written down, not accidental.
Negative · Internal

Weaknesses

Where the system is being fought, not used
  • W1
    Theme bypass at scale. Only 4 of 31 screens use useTheme(). 27 import tokens directly. Light theme is effectively dead code — and the escape hatch ("we force dark anyway") is an excuse, not a design.
  • W2
    50+ hardcoded hex values in screens. #F2C30F, #60A5FA, #EF4444, #22C55E — each one a silent fork that breaks when the palette evolves.
  • W3
    Orphaned components. Toggle atom has zero imports — profile.tsx ships its own inline version. The kit is growing faster than it's being adopted.
  • W4
    Screens are monolithic. route-planning.tsx is 1,715 lines; navigation.tsx 1,257; route-preview.tsx 1,088. That's a strong signal that molecules like SettingRow, SectionTitle, StatTile aren't being used.
  • W5
    Missing primitives. No opacity scale, no glass/surface tint tokens fully adopted, no card-chrome primitive used consistently. The same "dark card with 16 px radius and a hairline border" is re-typed dozens of times.
Evidence: design-work/design-system-analysis.md lists 50+ hardcoded hexes, 30+ inline rgba calls, 3 custom toggle re-implementations and 20+ direct token imports across screens.
Positive · External

Opportunities

Where small moves unlock big consistency
  • O1
    A single "Surface" primitive. One <Surface variant="glass|solid|outline" elevation="sm|md"> would replace ~30 inline card styles and collapse the glass/frosted pattern into a token.
  • O2
    Lint as the forcing function. ESLint rules banning hex literals and raw rgba() in app/** would stop drift at the PR boundary — the cheapest durable fix available.
  • O3
    Style-factory pattern, documented once. navigation.tsx already uses useMemo(() => createThemedStyles(colors)) — canonise that pattern, put it in a snippet, migrate the remaining 27 screens mechanically.
  • O4
    A component catalog inside the app. A hidden "/design" route that renders every atom/molecule against both themes would catch drift visually and double as onboarding for new engineers.
  • O5
    Mia persona as a design cue, not just copy. Persona branching exists in logic but the visual system doesn't mode-shift. A persona-aware accent-tint could make the "knowledgeable friend" voice visible at a glance.
Negative · External

Threats

What will hurt if left alone
  • T1
    Drift compounds. Every hardcoded colour today is a merge-conflict, regression or a11y bug tomorrow. At 50+ instances, the cost of a palette change is already non-trivial.
  • T2
    Safety semantics can leak into decoration. With red/amber/green hexes floating through screen code, nothing structurally prevents a "green stats card" from reading as a safety signal to a glancing cyclist.
  • T3
    Light mode becomes a liability. Shipped-but-broken light tokens are worse than no light mode. An OS-driven switch will expose the forked colours as visual bugs.
  • T4
    Component sprawl without governance. Mia, tiers, badges, streaks and community each added their own organisms. Without a retirement lane, the kit turns into a museum.
  • T5
    HUD readability regressions are invisible in staging. "Glanceable at 25 km/h" is a principle, but there's no contrast audit or outdoor-sunlight QA check gating releases.

Core diagnosis

The design system isn't under-built — it's under-trusted. Fix adoption and governance, and the score jumps from 6.4 to 8+ without shipping a single new token.

02

Recommendations

Ordered by ROI

Each item names the problem it solves, the argument for fixing it, and the first concrete step. Ranked by impact ÷ effort. Nothing here requires a design refresh — only governance and a handful of small primitives.

Priority01

Make the design system the only legal source of colour.

The single biggest drag on consistency is that raw hex values and rgba() literals ship inside screen code. Every one of them is a future palette-change bug and a potential safety-semantics leak. This is a governance problem, not a design problem — and it's cheap to fix once, expensive to fix forever by hand.

Add an ESLint rule banning /#[0-9a-f]{3,8}/ and /rgba?\(/ in apps/mobile/app/**. Allow an exception list for the three tokens/*.ts files. Block merges on violations.
Impact · High Effort · S P0
Priority02

Canonise the style-factory pattern and migrate screens mechanically.

StyleSheet.create() is static, which is why engineers reach for direct token imports — and that's how light theme gets killed. navigation.tsx already shows the correct pattern: const styles = useMemo(() => createThemedStyles(colors), [colors]). Document it, ship a codemod, move on.

Write a jscodeshift script that: (1) wraps each StyleSheet.create in a createThemedStyles(colors) factory, (2) rewrites direct darkTheme.X refs to colors.X, (3) inserts the useTheme() + useMemo pair. One screen a day = three weeks to full theme compliance.
Impact · High Effort · M P0
Priority03

Introduce a <Surface> primitive and delete card chrome everywhere.

The "dark card, 16 px radius, hairline border, optional glass tint" pattern is re-typed in most of the large screens. A single primitive — <Surface variant="solid | glass | outline" elevation="sm | md | lg"> — collapses dozens of style objects and makes the glass/frosted look a token instead of a folklore.

Extend the existing Card.tsx atom (already shipped per the audit) with an elevation prop mapped to shadows tokens. Add a Storybook-style catalog entry. Codemod finds borderRadius: radii.xl + backgroundColor pairs and replaces with <Surface>.
Impact · High Effort · M P1
Priority04

Ship a live component catalog at /design (dev-only).

The team has 25 atoms and 29 organisms with zero visual index. New engineers find components by grep, which is how inline toggles get born. A hidden catalog route — rendered against both themes and both personas — makes the kit discoverable and doubles as a visual regression target.

Add app/design/index.tsx behind __DEV__. For each atom/molecule, render a card of every variant/state. Tag each with its import path. Add screenshots to CI via Detox or a snapshot test — failures catch drift before users do.
Impact · Medium Effort · M P1
Priority05

Add a contrast + HUD-readability gate to CI.

"Glanceable at 25 km/h" is a beautiful principle with no teeth. The HUD renders distances, ETAs and hazard pills over a live map — the one place where a contrast regression is genuinely dangerous. Automate the check.

Write a test that renders NavigationHUD and HazardAlertPill against three map-tile samples (bright, mixed, night), runs a WCAG contrast-ratio calculation on rendered text/background pairs, and fails below 4.5:1 for body, 3:1 for large. Gate releases on it.
Impact · High Effort · M P1
Priority06

Decompose the three monolith screens.

route-planning.tsx (1,715 lines), navigation.tsx (1,257) and route-preview.tsx (1,088) are where the system most visibly fails. They're also the safety-critical surfaces where regressions hurt most. Smaller files force reuse and make review actually possible.

Extract PlannerHeader, DestinationPanel, NearbyPoiStrip from route-planning. Extract RoutePreviewBottomSheet, ElevationBlock, RiskBreakdownBlock from route-preview. Target: every screen file < 400 lines.
Impact · Medium Effort · L P2
Priority07

Retire the orphans — Toggle, unused tiers, stale molecules.

Either ship the Toggle atom into every setting row, or delete it. A component kit without a retirement policy grows into a museum — which is how three custom toggles ended up in profile.tsx when the canonical one was sitting unused in the atoms folder.

Quarterly sweep: any atom/molecule with grep count of 0 in app/** goes on a "ship or sink" list. Two-week grace period, then delete. Automate the usage-count report.
Impact · Medium Effort · S P2
Priority08

Make the safety palette non-bypassable.

The principle "safety is semantic, never decorative" is honoured in the token file and violated in the screens — every hardcoded #22C55E is a latent safety-signal leak. Structurally prevent it.

Move safetyColors behind a useSafetyColor(level) hook that logs a telemetry event every time it's called from outside a hazard/risk/route context (via a call stack check in dev). Export only through this hook from the public design-system index. Raw imports flagged at review.
Impact · High Effort · M P2
Priority09

Codify a density + hit-target audit.

The spacing scale exists; the 44 px minimum hit target is stated. Whether screens honour both is unverified. Given that users interact one-handed, at speed, with gloves, the cost of a 36×36 icon button is real.

Add a Detox integration test that walks every screen, measures bounding boxes of Pressable, Button, IconButton and fails any under 44×44. Pair with a per-screen density snapshot so reviewers see info-per-viewport trends.
Impact · Medium Effort · M P3
Priority10

Commit to dark-only, explicitly — or finish light mode.

Light tokens exist, light theme is effectively unreachable, and the app forces dark during the safety-critical path. That's a half-built option masquerading as a feature. Either delete the light tokens and own the decision ("dark, always — cyclist glare reduction"), or finish the migration so the OS-level switch works. The worst option is the current one.

Decide in a 30-minute design meeting. If dark-only: delete lightTheme export, remove the useColorScheme() branch, document the rationale in design-context.md. If both: ship Recommendation 02 and audit every screen in light.
Impact · Medium Effort · S P3
03

Refinement — smart, friendly, safe

Aesthetic direction

The current system reads as competent. To read as refined and smart while preserving the friendly, protective voice, the moves aren't a rebrand — they're micro-decisions in type, colour, surface and motion. Each one below is argued, and each one is a Tweak you can toggle live in the panel at the bottom-right.

01 · Typography

Pair a humanist display with a monospaced data voice.

Montserrat is geometric and a touch generic — it says "app", not "thoughtful companion". A humanist serif (Fraunces, Source Serif) at display sizes reads as editorial and trustworthy, while keeping Roboto Mono for data keeps the glanceable, engineered feel that safety requires. Warmth up top, precision at the numbers.

Move: Serif only at text3xl and above; body stays Roboto. Mono stays mono. Try the "Serif display" variant in Tweaks.
02 · Colour

Warm the neutrals; cool the accent.

Pure #111827 neutrals feel clinical. Nudge the dark surfaces two steps toward a warm slate (hint of brown, oklch(0.17 0.02 60)). It reads as a leather-bound notebook, not a server room — instantly friendlier without touching the safety palette. Keep yellow as signature; reserve red/amber/green strictly for semantic safety.

Move: Tweak "Warm neutrals" shifts bg by +5° hue. Notice how the yellow feels calmer against it.
03 · Surface

One elevation system, softly used.

Right now the hierarchy is communicated by hairline borders. Refined interfaces use gentle elevation: inner shadow on lowered surfaces, soft ambient shadow on cards, a 1 px top highlight for the accent cards. It signals craft without adding noise — and it's what distinguishes "app" from "tool".

Move: A <Surface> atom with elevation="inset | flat | raised". Toggle "Soft elevation" to preview.
04 · Radius & density

Slightly larger radii, slightly more air.

16 px is safe; 20 px feels held, considered, human. Combined with +8 px of vertical breathing room between sections, the interface gains authority — it's not trying so hard. Dense UIs read as busy; airy UIs read as confident.

Move: Radius slider + density toggle (cozy/default/airy). Watch the cards "settle".
05 · Motion

Motion as reassurance, never as flourish.

Friendly products feel alive but not busy. A 250–300 ms soft-spring on surface entrance, a subtle 1–2 px settle on focus, a slow pulse on safety glyphs only — and nothing moves during active navigation. This is the difference between "playful" (distracting at 25 km/h) and "warm" (calming in hand).

Move: Adopt spring easing tokens (stiffness 180, damping 22). Turn "Ambient motion" on to see them.
06 · Accent discipline

Treat yellow as punctuation, not paragraphs.

The yellow currently does a lot of work: active states, highlights, icons, dividers, tag colours. Refined systems use the brand colour sparingly — one accented element per viewport, usually the primary action. Everything else relaxes into warm greys. The yellow then carries real meaning when it appears.

Move: Slider "Accent intensity" dims non-primary yellow usage. Watch hierarchy snap into place.
07 · Copy & micro-voice

Warm the labels, tighten the data.

"Route Preview" is correct. "Your safer route" is friendlier without losing authority. Swap title-case UI labels for sentence-case; use first-person-plural where it fits ("We found…"); keep units and numbers terse and monospaced. Voice carries more personality than any pixel.

Move: No tweak for this — it's a copy pass. But rename Route Preview → Your route as a test.
08 · Iconography

Duotone safety glyphs, neutral utility glyphs.

Ionicons are fine as utility. The safety set (hazards, risk, caution) deserves a bespoke duotone treatment — solid fill for the shape, 40% overlay for the pattern. This sub-language visually separates "information you can ignore" from "information you must act on", a distinction the glanceability principle demands.

Move: Commission 8–12 custom safety glyphs. Keep system icons for everything else.

Positioning call

Google owns utility. Strava owns performance. Komoot owns planning. Defensive Pedal's wedge is protective intelligence — the only one of the five that treats real-time cyclist safety as a first-class design language. Lean into it: "the route your partner would choose for you".

6-week path

A pragmatic order that front-loads governance (stops the bleeding), then mechanises migration, then adds the primitives that prevent the next round of drift.

Week1

Stop the bleeding.

Ship the lint rule (R1). Decide dark-only vs full light (R10). Publish the style-factory recipe (R2 doc). No more colour drift lands after this week.

Governance
Weeks2-3

Mechanise migration.

Codemod for theme adoption (R2). Retire orphans (R7). First <Surface> rollout on the three biggest screens (R3).

Migration
Weeks4-5

Add the guardrails.

Component catalog (R4). Contrast + HUD gate in CI (R5). Safety-palette hook (R8). Hit-target audit (R9). The system is now self-defending.

Week6

Split the monolith screens (R6). Update design-context.md with the new canonical patterns. Re-score — target ≥ 8.0/10.
